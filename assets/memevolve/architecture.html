Deepagent Chat Loop Integration

MemEvolve is integrated as a pluggable component. During execution, it provides key historical context to the LLM agent and absorbs the current execution's trajectory for ongoing learning.

1 Initialization
When the CLI is launched (`cli.py`), it reads the `MEMORY_PROVIDER` environment variable. If set (e.g. `lightweight_memory`), it instantiates the `MemoryAdapter` with the base model wrapper and storage path. The adapter loads the backing EvolveLab provider dynamically.
2 Context Injection (Pre-Query)
Before sending the user prompt to the LangGraph execution, the CLI calls `MemoryAdapter.provide_context(user_input)`. The adapter creates a `MemoryRequest` with `BEGIN` status, gets the retrieved memory items from the provider, and pre-pends them to the messages list as a system instruction containing the historical logs.
3 Agent Execution
The LangGraph agent runs the task, executing subagent calls and tools. If a tool needs memory context, it can invoke the model callable, which utilizes the adapted model.
4 Trajectory Ingestion (Post-Query)
Once the agent finishes execution and returns the final state, the CLI calls `MemoryAdapter.absorb_trajectory(user_input, messages, result)`. The adapter translates the message history into a standardized `TrajectoryData` object and sends it to the provider's `take_in_memory()`. The provider extracts learning points and updates its disk state.

The Self-Evolution Closed Loop

Rather than using a fixed, hardcoded memory strategy, the project implements the MemEvolve framework to autonomously write, compile, and validate new memory provider implementations based on execution logs.

Phase 1 Analyze Trajectories (`PhaseAnalyzer`)
Reviews previous execution logs (JSON files) inside the workspace to identify failure modes (e.g. where the agent forgot tool behaviors, repeated previous mistakes, or failed to recall crucial user context). Generates a structured analysis report.
Phase 2 Generate Architecture (`PhaseGenerator`)
Uses the analysis report to engineer a new memory provider codebase. It writes prompts specifying required classes, custom scoring rules, memory pruning mechanisms, or vector weights that address the failures.
Phase 3 Write Files (`MemorySystemCreator`)
Dynamically writes the new provider Python code to `src/EvolveLab/providers/{provider_name}.py` and registers the class in `src/EvolveLab/memory_types.py` (updating the `MemoryType` enum and `PROVIDER_MAPPING` mappings). This creates a completely new, runnable module.
Phase 4 Validate System (`PhaseValidator` / `SWEAgentValidator`)
Runs static checks (syntax, imports) and executes unit tests to verify that the newly created memory module satisfies the abstract base class `BaseMemoryProvider` and performs correctly on task suites.

Evolution Feedback Loop

Note the prominent orange link in the diagram: MemorySystemCreator → Concrete Providers. This represents a closed-loop coding cycle where the LLM framework edits its own library structure to evolve better memory algorithms.

CLI Commands Reference

Running Deepagent with Evolved Memory

# Start agent with the evolved lightweight memory provider set MEMORY_PROVIDER=lightweight_memory python main.py # Start agent with custom memory database storage directory set MEMORY_PROVIDER=lightweight_memory set MEMORY_STORAGE_DIR=./custom_memories python main.py

Executing the Evolution Pipeline

# Run the full evolution pipeline automatically (runs Phase 1 -> 4) python -m deep_agents_from_scratch.evolve auto-evolve gaia --provider lightweight_memory --num-rounds 3 # Run individual evolution steps manually python -m deep_agents_from_scratch.evolve analyze ./trajectories/ --provider lightweight_memory python -m deep_agents_from_scratch.evolve generate --creativity 0.5 python -m deep_agents_from_scratch.evolve create python -m deep_agents_from_scratch.evolve validate

Core Adapter Implementation snippet

class MemoryAdapter: def __init__(self, provider_name: str, model=None, storage_dir=None): provider_cls = import_provider_class(PROVIDER_MAP[provider_name]) config = dict(PROVIDER_DEFAULTS.get(provider_name, {})) if model: config["model"] = _ModelWrapper(model) if storage_dir: config["storage_dir"] = storage_dir self._provider = provider_cls(config=config) def provide_context(self, query: str) -> str: request = MemoryRequest(query=query, context="", status=MemoryStatus.BEGIN) response = self._provider.provide_memory(request) return "\n\n".join([m.content for m in response.memories]) def absorb_trajectory(self, query: str, messages: list, result=None) -> tuple[bool, str]: trajectory = [{"role": m.type, "content": m.content} for m in messages] td = TrajectoryData(query=query, trajectory=trajectory, result=result) return self._provider.take_in_memory(td)

EvolveLab Evolved Memory Inventory

EvolveLab holds the inventory of baseline and evolved memory systems. These can be selected via the adapter by specifying their registered names:

Lightweight Memory
Registered: `lightweight_memory`. A standard-library-only evolved memory provider that stores items locally as JSON and requires zero external vector search dependencies.
Cerebra Fusion Memory
Registered: `cerebra_fusion_memory`. High-performance system utilizing sentence-transformers embeddings and scikit-learn metrics for semantic recall.
Agent KB Provider
Registered: `agent_kb`. The baseline system used as a starting template during the evolution process, supplying basic list-based memory.
Voyager Memory
Registered: `voyager`. A skill-based iterative memory provider that retrieves codes and instructions dynamically based on similarity.
Dilu Memory
Registered: `dilu`. Implements a case-based reasoning framework to search and extract past task demonstrations.
Generative Memory
Registered: `generative`. Implements the Generative Agent architecture (decay, importance, and recency scores).
